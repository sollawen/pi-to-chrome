## 补充分析：这个愿景 vs 当前 MCP 工具调用的现实

上面的设计思路（"把 find_elements 的结果对象直接传给下游工具"）在概念上是清晰的，但在 MCP 工具调用的实际机制下，有一个关键问题：

### MCP 工具调用不是代码赋值

文档中的愿景是：
```
data = find_elements(page, something)
trace_css(data.resultList[2])   // 直接传对象引用
```

但 MCP 工具调用的实际流程是：
```
1. pi 调用 find_elements → 返回文本给 LLM
2. LLM 看到文本输出，理解结果
3. LLM 构造 JSON 参数调用 trace_css
```

LLM 拿到的不是变量引用，而是**文本描述**：
```
3. <button.btn-danger> "删除"
   selector: .btn-danger   [index: 2]
```

LLM 需要从这段文本中提取字段，自己拼出 JSON 对象：
```json
{
  "locator": {
    "selector": ".btn-danger",
    "index": 2,
    "tag": "button",
    "text": "删除"
  }
}
```

这比直接写 `{ "selector": ".btn-danger", "index": 2 }` 繁琐很多。嵌套对象层级越深，LLM 出错概率越高。

### find_elements 已经能构建唯一 selector

看 find-elements.ts 的 `buildSelector` 函数——它会尝试 4 种策略（id → tag.class → ancestor path → nth-child fallback）来生成唯一 selector。

所以如果 LLM 按推荐流程先调 find_elements，返回的 selector **通常已经是唯一的**，直接传给 trace_css 就能用。多匹配问题主要发生在 LLM 跳过 find_elements、直接手写 `.card` 这种泛 selector 的场景。

### tag/text 联合校验的实际价值

| 场景 | 发生概率 | tag/text 校验有帮助？ |
|------|---------|---------------------|
| find_elements → 下游工具，间隔 < 2 秒 | 90%+ | 不需要，DOM 不会变 |
| 跳过 find_elements，直接用 selector | 常见 | 拿不到 tag/text，无法利用 |
| DOM 在两次调用之间发生变化 | 极少 | 有帮助，但 DOM 大变连 selector 都可能失效 |

为了一个极少发生的场景，每次调用都增加复杂度，性价比不高。

### 统一返回结构 `{ code, msg, number, resultList }` 的价值

这是个好想法，但它和"多匹配定位"是**两个独立问题**。当前 trace_css 返回 `{ element, cssRules }`，check_layout 返回 `{ element, layout }`，结构各不相同。统一返回结构值得单独做，但不应该和 index 方案捆绑——改动涉及所有工具的 details 字段，风险和范围都大了一级。

### 结论

上面的愿景（数据驱动、多属性联合定位、统一返回结构）方向是好的，可以作为**中长期目标**。但当前最紧迫的痛点——"selector 匹配多个元素时报错拒绝工作"——用一个简单的可选 `index` 参数就能解决：

```
LLM → trace_css({ selector: ".card" })
      → 报错："匹配了5个，用 index 参数"
LLM → trace_css({ selector: ".card", index: 2 })
      → 成功
```

2 次调用，LLM 只需加一个数字。改动面小，风险低，向后完全兼容。
