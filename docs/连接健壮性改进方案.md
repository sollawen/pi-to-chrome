# pi-to-chrome 连接健壮性改进计划

## 问题诊断

当前 TCP 连接正常（ESTABLISHED），但 `getActivePage()` 返回"无法获取当前页面"。
原因是代码中存在 **7 个空 `catch {}`**，错误信息被全部吞掉，无法定位真正失败点。

## 审查备注（2025-05-25）

以下问题已在计划中修正：
1. 断线回调中无 pi `ctx`，通知用户改用 `pi.sendMessage()`（非 `ctx.ui.notify`）
2. `ensureConnection()` 是在 `getActivePage()` **之前增加**的前置检查，不是"代替"
3. 断线回调中增加 `toolNames.length` 守卫，避免无意义操作

## 改进分 4 个层面

### 2. 断线感知 — `browser.on('disconnected')`

**文件：`core/browser.ts`**

新增：
- 连接建立后注册 `browser.on('disconnected', callback)`
- 回调中：将 `browser` 置 `null`，不做任何 CDP 请求（避免竞态）
- 导出 `onDisconnected(callback)` 注册接口，供 `index.ts` 注册清理回调

**文件：`index.ts`**

- 注册断线回调（注意：回调中无 pi `ctx`，只有 `pi` 可用）：
  1. **守卫**：`if (toolNames.length === 0) return;` 避免重连前重复触发
  2. **清理**：卸载工具（`pi.getActiveTools()` + `pi.setActiveTools()` 移除）、停止 console 监听
  3. **通知**：用 `pi.sendMessage()` 发送带 `display: true` 的自定义消息（不设 `triggerTurn`，避免触发无意义 LLM 调用）

```typescript
browser.onDisconnected(() => {
  if (toolNames.length === 0) return;

  consoleBuffer.stopListening();
  const allActive = pi.getActiveTools();
  const remaining = allActive.filter((name: string) => !toolNames.includes(name));
  pi.setActiveTools(remaining);
  toolNames = [];

  pi.sendMessage({
    customType: 'chrome-disconnected',
    content: '⚠️ Chrome 连接已断开！\n如需查看原因，请立即切换到 Chrome 查看 console 日志\n如需重新连接，请执行 /chrome-start',
    display: true,
  });
});
```

### 3. 健康检查 — 工具调用前的连接校验

**文件：`core/browser.ts`**

新增 `ensureConnection()` 方法：
- 检查 `browser !== null`
- 尝试 `browser.version()`（轻量操作）
- 失败则抛出明确错误：`'Chrome 连接已断开，请执行 /chrome-start 重连'`
- 可选优化：缓存最近一次检查结果（5 秒内有效），避免并发工具调用时重复 CDP 请求

**文件：`tool-registry.ts`**

每个工具的 `execute` 入口在 `getActivePage()` **之前**调用 `ensureConnection()` 作为前置健康检查：

```typescript
async execute(toolCallId, params, signal, onUpdate, ctx) {
  await ensureConnection();   // 新增：健康检查（失败则 throw 明确错误）
  const page = await getActivePage();  // 保留：获取活动页面
  return tool.execute(page, params, { consoleBuffer });
}
```

这样用户看到的不是模糊的"无法获取当前页面"，而是清晰的"连接已断开"提示。

---

### 4. 断线提示（实时调试入口）

**目标：提示用户立即切换到 Chrome 查看 console 日志，定位断线原因。**

**文件：`index.ts`**

断线回调已在第2步中完整定义（清理 + `pi.sendMessage()` 通知）。此步为实时调试提示的最终确认。

**注意：自动重连暂不实现。** 在日志数据积累充分、清楚断线原因之前，自动重连可能陷入循环断连，掩盖真正问题。后续根据日志评估是否需要。

## 注意事项

### 竞态条件防护

`disconnected` 事件触发时，browser 可能还处于半关闭状态。回调中：
- 避免再次调用 `browser.pages()` 等操作
- 直接标记状态，不做 CDP 请求
